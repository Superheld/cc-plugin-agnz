# Code-Review agnz — Stand 0.11.9 (2026-06-20)

Vollständiger Review über die gesamte Codebase (~5.900 LOC Quellcode). Durchgeführt
mit einem Kern-Read (`lib/loop.mjs` direkt) plus vier parallelen Subsystem-Reviewern
(MCP, Communication, Persistence/Config, Tools/Sandbox). Dieses Dokument hält die
Findings fest, damit die `file:line`-Referenzen nicht verloren gehen.

## 1. Architektur — wie sie heute tatsächlich aussieht

```
Claude Code (Parent)
   │  MCP stdio JSON-RPC (protocolVersion 2025-03-26)
mcp/server.mjs   ── 5 Tools, ZWEI Namespaces:
   │                  agent_*  : agent_start, agent_stop
   │                  thread_* : thread_send, thread_approve, thread_answer
   ▼
lib/loop.mjs (762 LOC) ── runThread(): LLM ↔ Tool-Loop, max 40 Turns
   ├─ sandbox.mjs        cwd-Lock + realpath + Policy (allow/ask/deny)
   ├─ tools/*            LS Read Grep / Edit Write / Bash / AskUser SendMessage Skill
   ├─ workspace-store    <cwd>/.claude/agnz/  (workspace.json + threads/)
   ├─ thread-index       ~/.claude/agnz/thread-index.json  (id → cwd, user-wide)
   ├─ messages-log       messages.jsonl + event-bus + notifier  (ADR 0002)
   ├─ agent-defs         .md+frontmatter Loader, buildToolPolicy
   ├─ profiles           LLM-Endpoints, user-wide
   └─ trace.mjs          <thread>.trace.jsonl — mit usage/Token-Logging (0.11.9)
```

Return-Channel ist **bereits file+hook-basiert** (Background-only): Ergebnisse kommen
über `SendMessage(to:"parent")` → `messages.jsonl` → `UserPromptSubmit`-Hook. MCP wird
faktisch nur für die *eingehenden* Kommandos benutzt.

## 2. Doku-Drift (CLAUDE.md ist auf 0.4er-Stand)

- Tools heißen `thread_send/approve/answer`, CLAUDE.md sagt durchweg `agent_send`.
- Tests existieren (`tests/`), CLAUDE.md listet "No tests" als Lücke.
- maxTurns default ist 40, nicht 20.
- `trace.mjs` hat einen dritten Entry-Typ `usage`.
- **ADR 0001, 0004, 0010 werden in CLAUDE.md referenziert, existieren aber nicht** in `docs/adr/`.
- **Zwei ADR 0006**: `0006-mcp-for-agents.md` + `0006-agentic-system-skills-memory-initialprompt.md`.
- `docs/next.md` ist leer.

## 3. Zustand

- Architektur sauber geschichtet, zero-dep wie beabsichtigt, sorgfältig kommentiert.
- **Test-Suite rot:** `node --test tests/` exit≠0 — `workspace-store.test.mjs:55`

## 4. Findings nach Schwere

Durchgehendes Thema: **kein JSON-Snapshot wird atomar geschrieben, mehrere
read-modify-write-Zyklen sind ungeschützt** — in einem System, das Nebenläufigkeit
als Design-Ziel hat.

### HIGH — Nebenläufigkeit & Datenintegrität

| # | Ort | Problem |
|---|---|---|
| 1 | `thread-index.mjs:43` | `registerThread` RMW auf globaler `thread-index.json`. Zwei parallele `agent_start` in verschiedenen Workspaces → ein Eintrag geht verloren → Thread permanent unauflösbar (`lookupThreadCwd → null`), obwohl Dateien existieren. |
| 2 | `threads.mjs:99` | `updateThread` nicht atomar. `writeThreadMeta` ist gechaint, aber `updateThread` liest *außerhalb* der Chain. Konkurrierend im Loop: `inboxCursor` (328), `sessionCommands` (715), `stopThread`→status. Lost-Update. Fix: `mutateThreadMeta(id, patchFn)` read+write in einem Chain-Link. |
| 3 | `messages-log.mjs:50` | Id-Reset auf `m000001` bei korruptem Last-Line. Neue Ids sind `<=` aller Cursor → Parent sieht diese Messages nie (`msg.id > cursor`), plus Id-Kollisionen. Fix: rückwärts zur letzten *gültigen* Zeile, nie resetten. |
| 4 | `messages-log.mjs:30` | `nextMessageId` liest die ganze Datei pro Append → O(n), quadratisch übers Log. Hook liest dieselbe Datei bei jedem Prompt. Keine Rotation. Fix: letzte Id in-memory cachen unter dem Mutex. |

### HIGH — Sandbox

| # | Ort | Problem |
|---|---|---|
| 5 | `sandbox.mjs:80` | Symlink-Escape über lexikalischen Tail. `resolveNonExistent` realpath't nur den existierenden Vorfahren, hängt den Rest lexikalisch an. Wird ein Zwischensegment nachträglich Symlink → Check zeigt auf falschen Pfad. Fix: nach mkdir/write finales Ziel re-realpathen + Check wiederholen. |
| 6 | `sandbox.mjs` + `Bash.mjs` | Bash umgeht die Sandbox komplett. cwd-Lock gilt nur für File-Tools; `/bin/sh -c` hat keine Pfad-Restriktion. `cat ../../etc/passwd`, `ln -s / escape` etc. trivial, sobald Bash allow/approved. **By-design, aber die Garantie "cannot escape cwd" ist mit Bash falsch.** Ehrlich dokumentieren oder OS-Level-Sandbox (`sandbox-exec`/bubblewrap). |

### MED

- `server.mjs:322` — `OUTCOME_SCHEMA` fehlt `"queued"`. `thread_send` gibt `queued` zurück; strikter 2025-06-18-Client lehnt die Antwort heute ab. **Echter Bug.**
- `jsonrpc.mjs:208` — `id:null` als Notification fehlklassifiziert → Client hängt. Nur *absent* (`!("id" in msg)`) ist Notification.
- `Edit.mjs`/`Write.mjs` — nicht-atomare Writes + RMW-Race. Fix: tmp+rename, per-path Mutex.
- `_lib.mjs formatMessages` — Prompt-Injection vom Sub-Agent in den Parent. Message-Bodies unsanitisiert in Claudes Context; `[agnz] N new messages`-Header forgeable. Fencing fehlt.
- `Bash.mjs:69` — Timeout killt Prozessgruppe nicht (SIGKILL an Shell reapt Grandchildren nicht); Timeout-Timer wird im Normalpfad nie gecleart. Fix: `detached:true` + `kill(-pid)`.
- `openai-compatible.mjs:96` — keine Response-Validierung. `choice.message` verbatim zurück. Gerade bei lokalen Modellen sollte `role`/`tool_calls`-Struktur garantiert werden.
- `threads.mjs:145` — `appendMessage` macht extra Full-RMW nur um `updatedAt` zu bumpen — häufigster Meta-Write, Hauptkandidat für Race #2.
- `workspace-store.mjs` / `thread-index.mjs` / `profiles.mjs` — kein atomic rename auf irgendeinem Snapshot-Write; partial write → harter `JSON.parse`-Fehler.
- `thread-index.mjs` — `stopThread` forgettet den Index nie, kein Prune → unbegrenztes Wachstum mit Stale-Einträgen.
- `event-bus.mjs:88` — Handler auf Name *und* `"*"` bekommt Message doppelt.

### LOW

- `loop.mjs:486-495` — verwaister Kommentarblock dokumentiert nicht mehr existierende Funktionen (eager skill-load, ADR-0009-Bash-Resolver).
- `loop.mjs:636` — `collectClaudeMds`-Docstring sagt "scan from cwd upward to home", Code geht nur cwd + visited subdirs nach unten.
- `agent-defs.mjs` — Tippfehler in `disallowedTools:` denied nichts (bleibt `ask`). Footgun. Auch: keine Validierung dass Policy-Einträge in `availableTools` sind; Sequence-Elemente werden nicht auf String geprüft.
- `Grep.mjs:94` — ReDoS: user-pattern ohne Backtracking-Schutz blockt den Single-Thread-Eventloop → stallt alle Agents.
- `Bash.mjs:92` — `exit_code:null` (SIGKILL) wird als `isError:false` gemeldet.
- `Read.mjs`/`Grep.mjs` — `split("\n")` lässt `\r` bei CRLF stehen; inkonsistent zu Edit/Skill die normalisieren.
- `openai-compatible.mjs:113` — `listModels` ohne Timeout (anders als `chat`).
- `LS.mjs:50` — Symlink-Ziele werden nicht aufgelöst/geflaggt (kein Escape, aber blind).
- `profiles.mjs:57` — `normaliseProfile` nur bei `add`, nicht bei `load` → handeditierte profiles.json failed erst zur Request-Zeit.
- `server.mjs` — `formatOutcome` + halbes `OUTCOME_SCHEMA` evtl. dead code im Background-only-Modell (prüfen ob importiert).

## 5. Empfehlung — Reihenfolge

1. **Concurrency-Paket als ein Arbeitspaket:** `mutate`-Primitive + tmp+rename als
   Konvention quer durch `thread-index`, `workspace-store`, `messages-log`,
   `Edit`/`Write`. Räumt #1, #2, #3 und Edit/Write-Race in einem Rutsch.
2. Sandbox-Garantie ehrlich machen (#5/#6 — härten oder dokumentieren dass Bash = Bypass).
3. `queued`-Schema fixen (echter Client-Bug).
4. Test-Suite grün machen, CLAUDE.md/ADR-Drift bereinigen.

> Hinweis: Mit dem geplanten MCP→CLI-Pivot fallen #server/jsonrpc-Findings (queued-Schema,
> id:null) ggf. weg bzw. ändern Form. Concurrency- und Sandbox-Findings bleiben relevant.
